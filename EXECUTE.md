# 工程辩证执行细则

`SKILL.md` 是唯一主契约。本文件只补充执行步骤和边界处理。

## 阶段 1：解析输入

识别用户提供的是候选方案还是工程目标。

若用户提供多个方案，保留每个方案核心主张。

若用户只提供目标，拆分 2 到 4 个工程立场，例如：

- 最小实现方案
- 可扩展架构方案
- 高可靠/高安全方案
- 渐进迁移方案

拆分时确保立场真的有差异，而不是同一方案换名字。

## 阶段 2：创建议题目录

生成英文目录名：

- 使用小写英文
- 用连字符分隔单词
- 保留工程目标核心含义
- 避免过长，通常 4 到 8 个英文词足够

目录冲突处理：

- 如果 `issues/{topic_en}/` 不存在，使用它
- 如果已存在，尝试 `issues/{topic_en}-v2/`
- 继续递增版本号，直到找到不存在的目录
- 不清理、不覆盖、不复用已有目录

## 阶段 3：初始化 dialectic_history.md

先将 `TEMPLATE.md` 复制为当前议题目录下的 `dialectic_history.md`。

复制后立即替换基本信息占位符：

- `{goal}`
- `{topic_en}`
- `{constraints}`
- `{stance_list}`

后续所有写入都通过替换唯一占位符完成。每次替换前先确认目标占位符存在且只出现一次。

## 阶段 4：串行调度子代理

每次调用子代理前确认：

- `dialectic_history.md` 已保存前一个角色的最新发言
- 本次调用的提示词包含工程目标、当前立场、当前阶段和文件路径
- 当前阶段目标明确

默认阶段：

| 阶段 | 目标 | 主 Agent 判断点 |
|------|------|----------------|
| 立场陈述 | 给出每个方案最强工程版本 | 是否覆盖收益、风险、验证方式 |
| 交叉质疑 | 暴露其他方案的关键工程风险 | 是否出现阻断性风险 |
| 收敛修正 | 回应质疑并尝试共识 | 是否坚持、让步、合并或转向 |
| 最终澄清 | 只处理剩余阻断点 | 是否仍需裁判判决 |

## 阶段 5：共识判断

主 Agent 在每轮结束后判断是否可以停止争论。

可以停止的信号：

- 子代理明确支持同一方案
- 分歧已被转化为边界条件或验证计划
- 某方案被充分击穿且无人继续维护
- 综合方案吸收了关键风险

不能停止的信号：

- 仍有未解决的安全、数据一致性、回滚、迁移或测试阻断点
- 子代理只是沉默，没有明确接受
- 方案依赖关键假设，但没有验证计划

## 阶段 6：最终方案

最终方案由主 Agent 输出并写入 `dialectic_history.md`。

必须覆盖：

- 最终决策
- 决策理由
- 适用边界
- 实施细节
- 验证计划
- 风险与未来预判
- 未决问题

如果没有完全共识，明确写明：

```text
未达成完全共识，但争论已充分，裁判做出如下判决。
```

## 阶段 7：验证

结束前检查：

- 新议题目录只包含 `dialectic_history.md`
- 文件中没有核心未替换占位符
- 每个立场至少完成立场陈述和交叉质疑
- 最终方案覆盖完整工程细节和未来预判

若发现当前运行误创建额外文件，删除当前运行误创建的额外文件。不要删除已有目录里的历史文件。

## 用户回复

向用户回复时不要复述完整记录，只报告：

- 记录文件路径
- 最终方案摘要
- 关键决策理由
- 实施要点
- 风险和未来预判
- 未决问题

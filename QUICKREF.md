# 工程辩证快速检查表

## 不变量

- 每次辩证只创建一个新目录：`issues/{topic_en}/`
- 新目录最终只允许有一个文件：`dialectic_history.md`
- 不创建议题 `README.md`、临时文件、日志文件或摘要文件
- 如果目录已存在，使用 `-v2`、`-v3` 等新目录，不覆盖旧内容
- 先复制 `TEMPLATE.md` 为 `dialectic_history.md`，后续只替换唯一占位符
- 篇幅是软约束，不做逐字计数

## 子代理硬约束

- 每个工程立场的观点都来自子代理
- 主 Agent 不得代写、补写、改写、润色或合成任何子代理发言
- 每次发言前必须实际调用对应子代理
- 每个子代理必须先读取最新 `issues/{topic_en}/dialectic_history.md`
- 子代理可以被其他方案说服，并应明确说明被说服的点
- 同一轮中相互依赖的子代理发言不得并行调用
- 子代理只输出发言，不写文件

## 默认流程

```text
阶段 1：立场陈述 -> 阶段 2：交叉质疑 -> 阶段 3：收敛修正 -> 可选最终澄清 -> 最终方案
```

## 内容质量检查

- 立场陈述要给出最强工程版本，而不是稻草人
- 交叉质疑要指出真实工程风险，不做泛泛价值判断
- 收敛修正必须说明坚持、让步、合并或转向
- 允许达成共识；这不是传统辩论赛
- 若未达成共识，裁判可以判决，但必须说明争论已充分
- 最终方案必须可执行、可验证、可回滚、可演进

## 最终方案必须覆盖

- 最终决策
- 决策理由
- 适用边界
- 实施细节
- 验证计划
- 风险与未来预判
- 未决问题

## 无效输出处理

子代理输出满足任一条件即无效：

- 空输出
- 要求写文件或声称已写文件
- 包含标题、解释、流程说明或非当前角色内容
- 请求用户补充信息
- 明显没有基于上下文发言
- 篇幅明显失控，影响讨论节奏

处理方式：重新调用同一角色子代理。主 Agent 不能替它修补。

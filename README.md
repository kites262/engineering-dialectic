# Engineering Dialectic Skill

`engineering-dialectic` 是一个面向软件工程决策的 opencode Skill。它通过多个子代理代表不同工程立场进行辩证讨论，帮助主 Agent 在架构、实现、迁移、测试、可靠性和安全性等问题上形成可执行的最终方案。

它不是传统辩论赛 Skill。目标不是让某一方赢，而是让方案在充分质疑和修正后达成共识；如果无法共识，则由主 Agent 在争论充分后裁判并输出工程决策。

## 适用场景

- 架构方案取舍：单体、模块化、微服务、插件化等。
- 实现细节选择：状态管理、缓存、并发、事务、权限、API 边界等。
- 工程目标拆解：用户只给目标，Skill 自动拆分多个可争辩的工程立场。
- 迁移与发布策略：渐进迁移、双写、灰度、回滚、监控和验收。
- 风险评估：复杂度、可靠性、安全性、可维护性、未来演进成本。

不适合用于没有明显方案分歧的简单实现任务，也不替代普通代码审查。

## 工作方式

默认流程分为三个阶段，并允许追加最终澄清：

```text
立场陈述 -> 交叉质疑 -> 收敛修正 -> 可选最终澄清 -> 最终方案
```

主 Agent 会先解析用户输入：

- 如果用户给出多个候选方案，保留各方案核心主张。
- 如果用户只给工程目标，自动拆分 2 到 4 个有实质差异的工程立场。

子代理会分别代表这些立场发言。它们可以坚持、让步、合并方案，或被其他立场说服。最终由主 Agent 判断是否已达成共识；若不能共识，则输出裁判判决。

## 输出内容

最终方案必须覆盖：

- 最终决策：采用哪个方案、合并哪些方案，或提出综合方案。
- 决策理由：为什么该方案优于其他方案。
- 适用边界：方案成立和不成立的条件。
- 实施细节：模块、接口、数据、状态、错误处理、权限、安全、性能、迁移、回滚。
- 验证计划：测试、监控、灰度和验收标准。
- 风险与未来预判：短期风险、中期演进、长期技术债或扩展路径。
- 未决问题：需要用户或团队继续确认的事项。


## 使用示例

只给工程目标：

```text
目标：为任务队列设计可靠的重试与死信机制
```

给出多个候选方案：

```text
方案 A：使用数据库表做任务队列
方案 B：使用 Redis Stream
方案 C：使用消息队列服务
```

更具体的工程决策：

```text
辩证分析：我们要在现有单体系统中引入插件机制，目标是支持第三方扩展，但不能破坏权限边界和升级稳定性。
```

## 产物

执行完成后，记录写入：

```text
issues/{topic_en}/dialectic_history.md
```

每个议题目录最终只保留 `dialectic_history.md`。生成的 `issues/` 目录是运行产物，默认不纳入本 Skill 仓库版本管理。

## 安装

将本目录放到用户级 opencode Skill 路径：

```text
~/.agents/skills/engineering-dialectic/
```

保存或更新 Skill 后，重启 opencode 使其重新加载。

# 工程辩证 Skill

## 权威规则

本 Skill 用于软件开发中的工程实践决策。主 Agent 担任主持人、裁判和方案整合者；多个子代理分别代表不同工程立场进行争辩，允许被说服、修正立场并走向共识。

若引用文件与本文件冲突，以本文件为准。

引用文件职责：

- `TEMPLATE.md`：最终 `dialectic_history.md` 的内容骨架
- `QUICKREF.md`：执行检查表
- `EXECUTE.md`：详细执行步骤和失败处理
- `README.md`：Skill 目录说明，不参与执行

## 适用边界

适用于用户要求解决软件工程中的开放性问题，例如：

- 多个技术方案取舍
- 架构边界、模块划分、接口设计
- 数据模型、状态管理、缓存、并发、事务、权限、安全方案
- 迁移策略、发布策略、测试策略、回滚策略
- 性能、可维护性、复杂度、可靠性之间的权衡
- 用户只给工程目标，需要自动拆分候选方案

不要在以下场景使用：

- 用户只是要求直接实现一个明确改动，没有明显方案分歧
- 用户只是要求解释已有代码或做普通 code review
- 问题可以通过一次事实查询或简单 grep/read 解决

## 输入解析

用户可能提供多个候选方案：

```text
方案 A：...
方案 B：...
方案 C：...
```

也可能只提供工程目标：

```text
目标：设计一个可靠的任务重试机制
```

解析规则：

- 若用户提供候选方案，主 Agent 必须保留每个方案的核心主张
- 若用户只提供目标，主 Agent 自动拆分 2 到 4 个有实质差异的工程立场
- 立场应覆盖不同工程价值，例如简单性、可扩展性、可靠性、性能、演进成本、安全性、交付速度
- 默认中文讨论；用户要求其他语言时才切换
- 若目标或约束严重不清，先问用户一个简短澄清问题；否则自动开始

## 产物边界

每次辩证创建一个新议题目录：

```text
issues/<topic_en>/
└── dialectic_history.md
```

目录规则：

- `<topic_en>` 使用小写英文短语，单词以连字符分隔，简洁表达工程目标
- 若目标目录已存在，使用 `<topic_en>-v2`、`<topic_en>-v3` 等下一个可用目录
- 新目录最终只保留 `dialectic_history.md`
- 禁止创建议题 `README.md`、`response.txt`、`result.json`、临时草稿、额外日志、额外摘要文件
- 不修改、不删除既有议题目录中的任何文件，除非这些文件是当前这次辩证刚创建的错误产物

## 角色边界

主 Agent 只能做这些事：

- 解析工程目标和约束
- 拆分或整理工程立场
- 创建议题目录和 `dialectic_history.md`
- 串行调度子代理
- 将子代理返回的原始观点替换到对应模板占位符
- 判断是否已经形成共识
- 必要时做最终裁判并整合解决方案
- 向用户输出最终工程方案、决策理由、实施细节和未来预判

主 Agent 严禁做这些事：

- 代替任一工程立场发言
- 补写、改写、合成、扩展、润色任何子代理观点
- 因为“答案显而易见”而跳过子代理调用
- 并行调用同一轮中相互依赖的发言子代理
- 在推理或执行中逐字、逐字符计数校验篇幅

子代理只能做这些事：

- 先读取当前议题目录下最新的 `dialectic_history.md`
- 站在被分配的工程立场上提出观点、质疑其他方案、回应质疑
- 在被说服时明确说明被说服的点，并修正自己的立场
- 只输出当前发言，不创建、不修改文件

## 辩证目标

工程辩证不是传统辩论赛，不以“击败对手”为最高目标。

优先级如下：

1. 找到工程上可执行、可维护、可验证的解决方案
2. 暴露被忽视的约束、边界、风险和未来演进成本
3. 让子代理在充分争论后可以互相说服并收敛
4. 若无法共识，由主 Agent 以裁判身份做最终判决

允许出现以下情况：

- 某个方案被完整否定
- 两个方案合并成折中方案
- 一个子代理被说服并转向支持另一方案
- 裁判发现所有候选方案都不足，提出综合方案

## 固定流程

默认执行 3 个阶段。每阶段是否继续由主 Agent 判断，但必须至少完成阶段 1 和阶段 2。

| 阶段 | 目标 | 子代理发言 |
|------|------|------------|
| 阶段 1：立场陈述 | 每个方案提出最强版本 | 每个立场 1 次 |
| 阶段 2：交叉质疑 | 指出其他方案的关键风险和盲点 | 每个立场 1 次 |
| 阶段 3：收敛修正 | 回应质疑，决定坚持、让步、合并或转向 | 每个立场 1 次 |

阶段 3 后若仍未达成共识，主 Agent 可以追加 1 轮“最终澄清”。若仍无法共识，主 Agent 主动裁判。

## 共识规则

满足以下任一条件，可以进入最终方案整合：

- 所有子代理明确支持同一方案
- 多数子代理支持一个方案，少数子代理的关键风险已被方案吸收
- 原候选方案合并成一个综合方案，且各方认可其关键边界
- 某个方案在风险、复杂度或约束上被充分击穿，各方不再维护它

若存在以下情况，不得假装共识：

- 仍有子代理指出未解决的阻断性风险
- 关键约束缺失，例如数据一致性、安全边界、回滚策略、测试策略
- 方案依赖未经验证的假设，却没有验证计划

## 子代理提示词模板

所有子代理提示必须包含：工程目标、当前立场、`dialectic_history.md` 路径、当前阶段目标、先读取上下文、只输出当前发言、不写文件。

### 立场陈述

```text
你现在是工程辩证中的【{stance_name}】代表。工程目标是“{goal}”。你的立场是：“{stance_position}”。当前阶段是【立场陈述】。请先读取 issues/{topic_en}/dialectic_history.md 获取上下文，并忽略未替换的 `{...}` 占位符。请给出该方案的最强工程版本：核心设计、适用前提、关键收益、主要风险和验证方式。请像资深工程师做方案评审一样发言，具体、克制、可执行。篇幅建议约 500 字以内，轻微超出可接受。只输出发言内容，不要标题、解释、文件操作或额外说明。
```

### 交叉质疑

```text
你现在是工程辩证中的【{stance_name}】代表。工程目标是“{goal}”。你的立场是：“{stance_position}”。当前阶段是【交叉质疑】。请先读取 issues/{topic_en}/dialectic_history.md 获取上下文，并忽略未替换的 `{...}` 占位符。请指出其他方案最关键的工程风险、隐藏成本、边界漏洞或未来演进问题；同时说明你的方案如何处理这些问题。允许承认对方的局部优点，但不要代替裁判总结。篇幅建议约 400 字以内，轻微超出可接受。只输出发言内容，不要标题、解释、文件操作或额外说明。
```

### 收敛修正

```text
你现在是工程辩证中的【{stance_name}】代表。工程目标是“{goal}”。你的立场是：“{stance_position}”。当前阶段是【收敛修正】。请先读取 issues/{topic_en}/dialectic_history.md 获取上下文，并忽略未替换的 `{...}` 占位符。请回应前面针对你方案的关键质疑，明确说明：哪些点坚持、哪些点修正、是否被其他方案说服、是否支持合并方案，以及最终建议的工程边界。篇幅建议约 400 字以内，轻微超出可接受。只输出发言内容，不要标题、解释、文件操作或额外说明。
```

### 最终澄清

```text
你现在是工程辩证中的【{stance_name}】代表。工程目标是“{goal}”。当前阶段是【最终澄清】。请先读取 issues/{topic_en}/dialectic_history.md 获取上下文，并忽略未替换的 `{...}` 占位符。只回答当前仍未解决的关键分歧：你是否接受当前共识方案？若不接受，阻断性风险是什么？需要加入什么边界或验证计划才可接受？篇幅建议约 250 字以内，轻微超出可接受。只输出发言内容，不要标题、解释、文件操作或额外说明。
```

## 裁判与最终方案

主 Agent 在以下情况下做最终裁判：

- 阶段 3 后已经形成共识
- 最终澄清后仍有分歧，但所有子代理已充分表达
- 用户需要明确执行方案，而继续争论收益很低

最终方案必须包含：

- 最终决策：采用哪个方案、合并哪些方案、或提出综合方案
- 决策理由：为什么它优于其他方案
- 适用边界：什么时候成立，什么时候不成立
- 实施细节：模块、接口、数据、状态、错误处理、权限、安全、性能、迁移、回滚
- 验证计划：测试、监控、灰度、验收标准
- 风险与未来预判：短期风险、中期演进、长期技术债或扩展路径
- 未决问题：需要用户或团队进一步确认的事项

裁判可以整合、取舍和判定，但不得伪造子代理已经同意的内容。若裁判主动判决而非共识形成，必须说明“未达成完全共识，但争论已充分”。

## 写入和验证规则

- 初始化时先将 `TEMPLATE.md` 复制为当前议题目录下的 `dialectic_history.md`
- 初始化后立即替换基本信息占位符
- 后续每次写入只替换当前阶段、当前立场对应的唯一占位符
- 每次启动子代理前，必须确保 `dialectic_history.md` 已包含之前所有最新内容
- 每次替换前检查目标占位符存在且唯一
- 全场结束后检查新议题目录只包含 `dialectic_history.md`
- 全场结束后检查没有残留必须填充的核心占位符

## 用户可见回复

结束后向用户输出：

- `dialectic_history.md` 路径
- 最终方案摘要
- 关键决策理由
- 实施要点
- 风险和未来预判
- 需要用户确认的未决问题

## 执行前必须读取

执行前读取：

- `QUICKREF.md`：检查表
- `TEMPLATE.md`：`dialectic_history.md` 内容模板

需要处理边界或失败场景时，再读取 `EXECUTE.md`。
